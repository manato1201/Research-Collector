// 第1章

問題定義と設計思想

1.1 問題を正確に定義する

「技術トレンドのキャッチアップに毎日30分かかっている。自動化できないか？」という一見シンプルな問いがある。しかし設計する前に問題を正確に分解する必要がある。

問題の分解
【曖昧な問い】 情報収集を自動化したい
      ↓ 分解

① どこから集めるか   → 情報源の定義（Zenn/Qiita/Unity/UE/CEDEC/論文）
② 何を集めるか      → フィルタ条件（タグ・キーワード・対象分野）
③ どこに蓄積するか  → ストレージ（NotebookLM・週次ノートブック）
④ どう活用するか    → ユースケース（チャット検索・週次レポート）
⑤ いつ・どう動かすか → 実行環境（GitHub Actions・タスクスケジューラ）
実務家の視点

要件が曖昧なまま実装を始めると、「動くけど使えない」ものができる。設計の7割は要件定義だ。コードを書き始める前に、必ずこの分解を行うこと。

1.2 制約の明確化

設計は制約の中で行う。制約は敵ではなく、設計の指針になる。

制約内容設計への影響
コスト無料または最小限有料API不使用・GitHub Actions無料枠
保守性一人で運用可能自動化・障害通知・再現性確保
拡張性情報源を追加しやすいコレクターをプラグイン的に追加可能
セキュリティAPIキー・認証情報を漏洩させないGitHub Secrets・.gitignore徹底

1.3 アーキテクチャの選択

3つの候補を検討し、GitHub Actionsを採用した。

案A: ローカルcron

✓ シンプル・低コスト
✗ PCが起動していないと動かない
✗ 障害通知が難しい

案B: AWS Lambda等

✓ 常時稼働・スケーラブル
✗ 費用がかかる
✗ 設定が複雑

案C: GitHub Actions（採用）

✓ 無料・コードと一体管理
✓ 障害はEmailで自動通知
✗ 実行環境に制約あり（X Server なし等）

// 第2章

データ収集のアルゴリズム

2.1 RSSフィードの仕組み

RSS（Really Simple Syndication）は、Webサイトが更新情報を配信するためのXMLフォーマット。

<?xml version="1.0" encoding="UTF-8"?>
<rss version="2.0">
  <channel>
    <title>Zenn - unity</title>
    <item>
      <title>Unity 6 新機能まとめ</title>
      <link>https://zenn.dev/xxx/articles/yyy</link>
      <pubDate>Mon, 03 Jun 2026 10:00:00 +0000</pubDate>
    </item>
  </channel>
</rss>

feedparser ライブラリで以下のようにパースできる。

import feedparser

feed = feedparser.parse("https://zenn.dev/topics/unity/feed")
for entry in feed.entries[:5]:
    print(entry.title, entry.link)
RSSHTMLスクレイピング
公式に提供されたデータHTML構造に依存（壊れやすい）
構造が標準化されているパース処理が複雑
利用規約上問題なしグレーゾーンの場合がある
軽量ページ全体取得のため重い

2.2 重複チェックのアルゴリズム

同じ記事を毎日NotebookLMに追加してしまう問題を解決する。

ナイーブな実装（問題あり）
seen_urls = []
if url not in seen_urls:
    seen_urls.append(url)
# O(n) の線形探索
# ファイルが肥大化する
ハッシュによる解決（採用）
import hashlib

def url_hash(url: str) -> str:
    return hashlib.sha256(
        url.encode()
    ).hexdigest()[:16]

seen: set[str] = load_seen()
if url_hash(url) not in seen:
    process(url)
# O(1) の定数時間探索
なぜURLそのままではなくハッシュを使うか

1. ストレージ効率: URLは可変長（50〜200文字）だが、ハッシュは固定16文字
2. セキュリティ: publicリポジトリでURLを公開したくない（一方向性）
3. 探索効率: PythonのsetによるO(1)検索

SHA256の特性: 衝突確率は約 1/2^64 ≈ 5.4×10⁻²⁰（実用上無視できる）

2.3 フォールトトレラントな収集設計

1つのフィードが失敗しても全体が止まらない設計が重要。

def run_daily():
    all_articles = []

    for collector_name, collector_fn in collectors.items():
        try:
            articles = collector_fn()
            all_articles.extend(articles)
            logger.info(f"{collector_name}: {len(articles)} articles")
        except Exception as e:
            # 失敗してもログを残して続行（raiseしない）
            logger.error(f"{collector_name} failed: {e}")

    process(all_articles)
設計判断の注意点

except Exception で広く捕捉するのは通常は悪い習慣。しかしバッチ処理では「部分的な失敗を許容してできるだけ多くを処理する」が正しい。重要なのは例外を握りつぶすのではなく、ログに残すこと。

非公式APIとの統合

3.1 Cookie認証の仕組み

NotebookLMには公式APIがない。notebooklm-py は以下の方法で実現している。

# ブラウザ（Playwright）でGoogleにログイン
# → Cookie（セッション情報）を storage_state.json に保存
# → 以降はそのCookieを使ってHTTPリクエストを送信

auth_json = os.environ.get("NOTEBOOKLM_AUTH_JSON", "").strip()
tmp = tempfile.NamedTemporaryFile(mode="w", suffix=".json", delete=False)
tmp.write(auth_json)
client = await NotebookLMClient.from_storage(path=tmp.name)
Cookieとは
HTTP Cookie = Webブラウザがサーバーから受け取り保存する小さなデータ
役割: 「このユーザーは認証済み」という状態をサーバーに伝える

Set-Cookie: session=abc123; HttpOnly; Secure; SameSite=Strict
             ^^^^^^^        ^^^^^^^^  ^^^^^^  ^^^^^^^^^^^^^^^^^
             値             JSから   HTTPS   クロスサイト制限
                           読めない  のみ

3.2 非公式ライブラリのリスク管理

リスク対策
仕様変更で突然動かなくなるエラー検知 + Issue通知で素早く気づく
セキュリティリスクOSSなのでソースを確認できる
メンテナンス停止フォークして自分でメンテする準備
実務の教訓

非公式ライブラリの使用は「技術的負債」の一種。利便性とリスクを天秤にかけて判断する。代替手段がない場合は依存を局所化することが重要（今回は nbklm/client.py に閉じ込めている）。

3.3 環境変数による秘匿情報管理

悪い例（コードにベタ書き）
client = NotebookLMClient(
    api_key="sk-ant-api03-xxxx"
)
# ← Gitに残る・公開されてしまう
良い例（環境変数から取得）

    "NOTEBOOKLM_AUTH_JSON", ""

CI/CDとしてのGitHub Actions

4.1 基本構造

name: Daily Research Collect

on:
  schedule:
    - cron: '0 21 * * *'   # UTC 21:00 = JST 6:00
  workflow_dispatch:        # 手動実行も可能

jobs:
  collect:
    runs-on: ubuntu-latest

    steps:
      - uses: actions/checkout@v4
      - run: python main.py --mode daily
        env:
          SECRET_KEY: ${{ secrets.SECRET_KEY }}
cronの読み方
分  時  日  月  曜日
0   21  *   *   *
↑   ↑   ↑
0分 21時 毎日（UTCで指定 → JST = UTC+9 → 翌朝6:00）

* はワイルドカード（毎〜という意味）

4.2 冪等性（Idempotency）

冪等性 = 何度実行しても同じ結果になる性質。CI/CDでは必須の考え方。

冪等でない（問題あり）
def add_article(url):
    notebooklm.add(url)
# 2回実行すると重複が発生
冪等（採用）
def add_article(url):
    if url_hash(url) not in seen:
        notebooklm.add(url)
        seen.add(url_hash(url))
# 何度実行しても同じ結果
なぜ冪等性が重要か

Actionsが途中で失敗して再実行した場合、冪等でないと重複データが生まれる。バッチ処理・CI/CD・データパイプライン設計では常に冪等性を意識すること。

4.3 Artifactによる状態の永続化

GitHub Actionsの各実行は独立した環境で動く。実行間でデータを引き継ぐためにartifactを使う。

# 復元（実行開始時）
- uses: actions/download-artifact@v4
  with:
    name: seen-urls
    path: .
  continue-on-error: true  # 初回は存在しないのでエラーを無視

# 保存（実行終了時）
- uses: actions/upload-artifact@v4
  with:
    name: seen-urls
    path: seen_urls.txt
    overwrite: true         # 前回のartifactを上書き

NotebookLMをAI外部脳として活用

5.1 RAGとNotebookLMの関係

RAG（Retrieval-Augmented Generation）の仕組み
質問
  → ベクトル検索で関連ドキュメントを取得  ← Retrieval
  → 取得したドキュメントをコンテキストとしてLLMに渡す ← Augmented
  → LLMがドキュメントを参照しながら回答生成 ← Generation
  → 回答に参照元URLが付く（ハルシネーション低減）

NotebookLMはRAGの実装であり、アップロードしたドキュメントが検索対象のコーパスになる。

Research CollectorとNotebookLMの関係

Research Collector = コーパスを自動的に更新し続けるシステム
NotebookLM = そのコーパスに対するRAGエンジン

毎日新しい技術記事・論文が追加される → 常に最新の知識ベースに対してAI検索できる

5.2 トークン消費ゼロの実現

通常のアプローチ（高コスト）

PDFを読み込んでClaudeに質問
→ 入力トークン: ドキュメント全文 × n
→ 高コスト・すぐに制限に達する

NotebookLM活用（コストゼロ）

URLをNotebookLMに登録
→ NotebookLMが内部でインデックス化
→ 質問はNotebookLMが処理（Googleのインフラ・無料）

5.3 週次ノートブック設計の背景

なぜ1つの大きなノートブックではなく週次に分けるか？

制約: NotebookLM Plusは1ノートブックあたり300件上限

1週間の収集件数: 約150件
3カテゴリ × 300件 = 900件/週
  → 週ごとにリセット = 300件を超えない設計

命名規則:
year, week, _ = datetime.now().isocalendar()
name = f"Game-Dev-Tech-{year}-W{week:02d}"
# 例: "Game-Dev-Tech-2026-W20"

6.2 publicリポジトリ公開時のチェックリスト

公開前チェックリスト
□ .gitignore に .env, storage_state.json を追加
□ コード内にAPIキーのベタ書きがないか確認
□ notebook_ids.py のIDを環境変数化
□ seen_urls.txt が含まれていないか確認
□ GitHub の Secret Scanning を有効化
□ リポジトリを private に設定（推奨）
重要な教訓

一度Gitにpushした秘匿情報は git rm しても履歴に残る。発覚した場合はAPIキーを即時失効させ、新しいキーを発行する。GitHubのSecret Scanningを有効にしておくと自動検知できる。

課題 1 — 基本

]

ヒント: zenn_qiita_collector.py を参考に実装する。

課題 2 — 応用

重複チェックの改善を考える

現在の実装はURLベースの重複チェックのみ。タイトルが類似した記事も除外したい場合、どのようなアルゴリズムが適しているか考察せよ。

検討すべきアルゴリズム

編集距離（Levenshtein距離）: 文字列の変換コストで類似度を測る。O(n²)の計算量。
TF-IDF + コサイン類似度: 単語の重要度ベクトルで類似度を計算。
埋め込みベクトル（Embedding）: LLMで文章を数値ベクトル化し意味的類似度を計算。精度最高だがコストがかかる。

課題 3 — 設計

自分の卒論テーマ向けにカスタマイズ

あなたの卒論テーマに必要な論文キーワードを設計し、paper_collector.py の ARXIV_QUERIES を修正せよ。
